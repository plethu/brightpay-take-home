Checkout till — mockup gallery

Three interaction models for the cashier till. See scratch/05-design.md for the why.

Same data in every mockup (SPEC pricing, GBP). Each renders one till; the floating dev-tools panel (bottom-right, marked “not product UI”) switches between the reachable states — empty basket, item added with offers, and a validation error — and toggles the theme. State and theme are also URL-driven (?state=error, ?theme=dark) so a given view is shareable. All three link the real wwwroot/app.css tokens, so light/dark and the savings/offer colours are the genuine article.

A — Two-column till

Operator console: scan + persistent price-list reference on the left, basket and a sticky total on the right. Best when the SKU list is genuinely consulted.

B — Receipt-centric

Single column, mobile-first. The basket reads top-to-bottom like a till receipt with the total anchored at the bottom; price list tucked into a reference panel.

C — Keypad grid

Tappable SKU tiles (price + offer on each face) as the primary add mechanism; text scan field retained as the keyboard / no-JS fallback. Most “physical till”.

D — Receipt + reference rail (converged)

B's vertical scan→total flow with a sticky total so the hero never scrolls away, plus a persistent price-list rail whose rows double as add controls (A's always-on reference + C's tap-to-add). Right IA, generic SaaS skin.
